# Full Project Acceptance Hard Gates

Use this skill when the user asks for **完整项目验收能力**: not a recap of one project result, but a repeatable way to decide whether a project is acceptable.

The skill must produce a hard-gated acceptance report. Every required area must be marked `PASS`, `CONDITIONAL`, `FAIL`, or `NOT VERIFIED` with evidence.

## Core rule

Do not accept a project just because one CI run is green.

A project passes only when:

1. the original requirement is converted into a complete requirement matrix;
2. each requirement is mapped to current implementation;
3. critical security and access-control behavior is verified separately;
4. generated test artifacts are inspected, not assumed;
5. probe runs are not confused with full-suite runs;
6. blocking gates have no failures.

## Evidence hierarchy

Use evidence in this order:

1. Actual executed test logs and generated artifacts for the target commit.
2. CI workflow runs, job status, and uploaded artifacts.
3. Generated summaries committed by automation.
4. Source code, migrations, config, and scripts.
5. Static or contract guard scripts.
6. Product/API/operation documentation.
7. Manual smoke UI.
8. User claim.
9. Assistant-generated summary.

Assistant-generated summaries are never test evidence.

## Required output

The final report must include:

```text
Scope
Target commit/branch
Original requirement source
Requirement matrix
Hard gate table
Evidence table
Artifact table
Gap table
Final verdict
Caveats
```

## Status vocabulary

Use only these statuses:

```text
PASS          Requirement is implemented and evidence supports it.
CONDITIONAL   Mostly acceptable, but evidence is incomplete or probe-only.
FAIL          Required behavior missing or contradicted.
NOT VERIFIED  Evidence was unavailable or not checked.
```

## Gap severity

```text
P0 blocker      Cannot accept.
P1 conditional  Accept only with explicit caveat/follow-up.
P2 quality      Non-blocking improvement.
Evidence gap    Implementation may exist, but proof is missing.
Spec gap        Requirement wording needs decision.
```

Any P0 blocker means final verdict cannot be PASS.

# Hard gates

## Gate 0: Evidence provenance gate

Must check:

- target repository;
- target branch or commit;
- latest relevant PRs;
- whether evidence belongs to current code;
- whether a report is generated by tests/CI or by the assistant.

PASS requires:

- all evidence tied to a commit, run, artifact, or file path;
- assistant reports clearly labeled as review summaries only.

FAIL if:

- old audit conclusions are reused without re-reading current code;
- assistant-generated report is presented as test artifact;
- artifact source command/workflow is unknown.

## Gate 1: Original requirement coverage gate

Must check:

- original prompt/spec exists;
- every product requirement is converted into an atomic checklist item;
- no major requirement is skipped.

Required categories:

```text
architecture
runtime/deployment
data model
storage
API contract
workflow/state machine
RBAC/security
auth/session/replay
encryption/PII/masking
domain features
audit/notifications
backup/restore/operations
UI/manual validation
documentation
tests/CI/artifacts
```

PASS requires a requirement matrix with evidence for each major item.

FAIL if a final verdict is issued without a requirement matrix.

## Gate 2: Architecture and deployment gate

Must check:

- language/framework required by spec;
- database/persistence technology;
- local file/object storage behavior;
- Docker/build packaging;
- runtime assets copied into image;
- external dependency posture.

Evidence sources:

```text
go.mod
server/bootstrap files
migrations
Dockerfile
docker-compose or CI Docker job
runtime acceptance logs
```

FAIL if required runtime architecture is absent or build/runtime cannot start.

## Gate 3: Data model and storage gate

Must check:

- migrations define all required entities;
- foreign keys/unique constraints/check constraints enforce core rules;
- version/history/audit tables exist when required;
- source-of-truth fields are clear;
- compatibility/plain fields are not treated as protected source of truth.

PASS requires schema and repository/write-path alignment.

FAIL if required data cannot be persisted or core constraints are missing.

## Gate 4: Authentication, session, and replay gate

Must check:

- password hashing;
- failed-login lockout;
- JWT/session creation;
- inactivity timeout;
- token revocation/blacklist;
- request timestamp freshness;
- replay guard/request ID behavior;
- negative tests or contract checks.

Evidence sources:

```text
auth middleware
session schema
jwt blacklist schema
request replay schema
auth tests/API tests
```

FAIL if authentication works only on happy path and replay/session invalidation is unverified.

## Gate 5: Protected data and PII encryption gate

Must check every sensitive field named by the spec.

For each field, verify:

```text
classification
schema column
write path
read/decrypt path
API exposure/masking
test or static contract
```

For encryption claims, verify:

```text
algorithm/helper exists
key source exists
ciphertext prefix/format exists
write path stores ciphertext
read path decrypts only where allowed
plain compatibility field is not source of truth
```

Typical required protected fields include:

```text
password verifier/hash
username or login identity
display name
PII metadata
document title or filename when classified sensitive
redaction coordinates
redaction reason
annotation comment
notification message
audit source IP
audit metadata
```

FAIL if the spec requires encrypted storage but the current write path stores plaintext as source of truth.

## Gate 6: RBAC and field visibility gate

Must check:

- all roles required by spec;
- allowed actions by role;
- forbidden actions by role;
- object-level access;
- sensitive fields visible/hidden by role;
- ciphertext/helper fields never serialized;
- positive and negative tests.

Required matrix:

```text
Role | Allowed actions | Forbidden actions | Visible fields | Hidden fields | Evidence
```

FAIL if only positive access paths are tested.

## Gate 7: Workflow and immutability gate

Must check:

- all required statuses exist;
- valid transitions are enforced;
- invalid transitions rejected;
- terminal/finalized state is immutable;
- history rows written;
- audit rows written;
- tests cover valid and invalid paths.

FAIL if terminal immutability or invalid transition rejection is not enforced.

## Gate 8: Domain feature gate

Must check every domain feature in the original requirement.

For each feature verify:

```text
endpoint/command
service logic
storage mutation
side effects
error handling
positive tests
negative tests
artifact/log evidence
```

Examples:

```text
upload/import
PDF validation
versioning/rollback
redaction staging
redaction burn-in
Bates numbering
comparison/diff
finalization
notifications
backup/restore
```

FAIL if a core domain feature is only documented but not implemented.

## Gate 9: Audit, notification, and configuration gate

Must check:

- mutating actions write audit events;
- audit metadata is protected if sensitive;
- notifications are created for required events;
- mention notification behavior if required;
- notification read acknowledgement;
- templates/config/workflow dictionaries are configurable when required;
- Admin-only management paths are protected.

FAIL if audit exists only for some mutating flows without documented exception.

## Gate 10: API contract, errors, and pagination gate

Must check:

- OpenAPI/Swagger generation;
- route coverage;
- status code coverage;
- request/response schemas;
- authentication annotations;
- uniform error envelope;
- pagination default and hard max;
- contract tests/static checks.

FAIL if generated API docs are missing for required routes or error contract is inconsistent.

## Gate 11: Backup, restore, and operation gate

Must check:

- scheduled backup behavior;
- logical dump behavior;
- filesystem snapshot behavior if required;
- restore endpoint or procedure;
- PITR documentation if required;
- config for schedule/paths;
- tests or contracts for scheduled backup.

FAIL if backup is mentioned in docs but no runnable/scheduled mechanism exists.

## Gate 12: Local test entrypoint gate

Must check every local test script.

For `run_tests.sh`-style scripts, record:

```text
normal command
probe command
required services
output directory
summary files
HTML/report files
logs
exit behavior
stage list
```

Hard distinction:

```text
full local run    = complete local acceptance evidence
probe/contract run = entrypoint/report-generation evidence only
```

FAIL if probe evidence is reported as full-suite evidence.

## Gate 13: Full regression gate

Must check full regression separately from local tests.

Required checks:

- full regression script exists;
- stage list is known;
- summary JSON/Markdown generated;
- CI or post-merge workflow runs it;
- Docker/runtime acceptance included when required;
- artifacts or committed summaries are available.

Typical full regression stages:

```text
prepare_workspace
swagger_contract
swagger_route_coverage
scheduled_backup_contract
metadata_storage_contract
gofmt
go_vet
go_test_race
shell_syntax
docker_build
docker_acceptance
```

PASS requires generated summary showing overall passed or equivalent.

## Gate 14: CI workflow and artifact gate

Must check:

- workflow triggers;
- path filters;
- skipped jobs;
- job conclusions;
- artifact upload paths;
- retention;
- `if: always()` behavior for failure evidence;
- whether artifacts are downloadable;
- whether the downloaded artifact matches the claimed command.

FAIL if the relevant job was skipped but the report calls CI passed for that area.

## Gate 15: Manual UI/smoke gate

If a simple UI/manual validation surface is required, check:

- UI file exists;
- app serves it;
- Docker/runtime packages it;
- it can call critical API flows;
- scope is clear: manual test aid vs production UI.

FAIL if user-facing/manual UI is required but neither implemented nor explicitly out of scope.

## Gate 16: Documentation gate

Must check:

- setup/run docs;
- configuration docs;
- API docs;
- backup/restore/PITR docs;
- security model docs;
- test/acceptance docs;
- known limitations.

FAIL if docs contradict current code on critical behavior.

## Gate 17: Final verdict gate

Before final verdict, produce:

```text
hard gate table
requirement coverage table
test evidence table
artifact provenance table
gap severity table
final pass/conditional/fail statement
```

Final PASS is allowed only if all P0 gates pass.

# Anti-false-acceptance checklist

```text
[ ] Original requirements reconstructed
[ ] Current code checked
[ ] Requirement matrix complete
[ ] Hard gates evaluated
[ ] Security checked separately
[ ] RBAC denied paths checked
[ ] CI skipped jobs inspected
[ ] Artifacts downloaded or parsed when available
[ ] Probe vs full-suite distinguished
[ ] Assistant report not used as test evidence
[ ] Caveats stated clearly
```

If any unchecked item affects a P0 gate, do not issue PASS.

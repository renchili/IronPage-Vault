# Docker Builder Workflow

IronPage Vault is built as one offline service image containing PostgreSQL, the Go API, migrations, local PDF dependencies, and the acceptance-only browser asset.

## Supported build path

The supported first build is:

```bash
bash scripts/deploy.sh
```

The deployer generates the installation configuration before invoking Compose. The Dockerfile requires generated build arguments for the application root and HTTP port, so a bare `docker build .` is not the supported deployment command.

After `.env` exists, the equivalent manual build is:

```bash
docker compose --env-file .env build ironpage
```

## Image stages

The Dockerfile uses two stages:

1. the Go builder resolves modules, generates Swagger, and compiles the server binary;
2. the runtime stage packages PostgreSQL, the binary, migrations, the canonical acceptance UI, and the entrypoint at installation-specific generated paths.

Runtime credentials, database identity, listener ports, and data paths are not stored as fixed image defaults.

## Module resolution

The repository build does not depend on a pre-generated Swagger package. The builder prepares it before compilation. Module resolution occurs in the controlled builder stage.

## Complete acceptance

Docker-backed acceptance is orchestrated by:

```bash
bash ci/docker_acceptance.sh
```

Complete retained evidence is generated by:

```bash
bash ci/run_full_regression.sh artifacts/regression
```

Both entrypoints generate installation configuration rather than relying on a fixed host port, container name, database identity, filesystem path, or credential.

# Backup and Recovery

IronPage Vault is designed for local air-gapped operation. Backup and recovery must use local volumes and local files only.

This document describes the current strict backup and restore semantics. It must not be read as a future-worker plan.

## Data categories

The system has two data categories that must be recovered consistently:

1. PostgreSQL data.
2. Local PDF binary files.

PostgreSQL contains users, sessions, document metadata, version metadata, workflow history, annotations, redaction metadata, Bates job metadata, audit logs, notifications, configuration entries, and backup job metadata.

The filesystem contains PDF binaries, transformed PDF versions, and local backup outputs.

## Docker volumes

The Compose file declares local volumes for PostgreSQL data, PDF storage, and backup outputs:

```text
ironpage_pgdata
ironpage_storage
ironpage_backups
```

Inside the container these map to:

```text
/var/lib/postgresql/data
/var/lib/ironpage/storage
/var/lib/ironpage/backups
```

## Current backup behavior

The Admin backup API creates a full local backup job when strict artifact creation succeeds.

The implementation path is:

```text
internal/app/backup_file.go -> internal/platform/backup_strict.go -> internal/platform/backup_exec.go
```

Strict backup requires both artifacts:

| Artifact | Required mode | Purpose |
| --- | --- | --- |
| PostgreSQL custom dump | `pg_dump_custom` | Database restore input |
| Filesystem tar snapshot | `tar` | PDF storage restore input |

If either artifact is missing or generated by a non-strict fallback mode, strict backup returns an error and the API must not report a restore-supported backup.

The response kind for a successful strict backup is:

```text
full_backup
```

The artifact manifest records:

```text
backup_id
created_at
database_dump_path
file_snapshot_path
database_dump_mode
file_snapshot_mode
restore_supported
```

## Operational prerequisites

The runtime environment must provide local backup tooling inside the container or host context used by the application:

- PostgreSQL dump tooling for creating custom database dumps.
- PostgreSQL restore tooling for restoring custom database dumps.
- `tar` for PDF storage snapshot and extraction.
- Writable local backup volume.
- Readable local PDF storage volume.

No backup or restore step may require cloud storage, remote services, or internet access.

## Restore behavior

Strict restore requires both paths:

```text
database_dump_path
file_snapshot_path
```

The strict restore path rejects empty or unreadable artifact paths. A successful restore must report:

```text
database_restore=pg_restore
file_restore=tar
```

## Recovery order

1. Stop writes to the application.
2. Identify the backup manifest and its database dump plus file snapshot paths.
3. Restore PostgreSQL data using the database dump artifact.
4. Restore PDF storage files using the tar snapshot artifact.
5. Start the application.
6. Verify `/healthz`.
7. Verify representative document metadata, version records, and file download paths.
8. Verify audit and notification records expected for the recovery point.

## Consistency requirement

Database metadata and PDF files must represent the same recovery boundary. If metadata points to a file path that was not restored, document download and version inspection will fail.

## Acceptance checks

- Admin can create a strict full backup.
- Backup job rows are queryable.
- The manifest records database and file artifacts.
- `restore_supported` is true only for strict restore-capable artifacts.
- Restore requires both database and filesystem artifact paths.
- No backup or restore step requires cloud storage or internet access.

## Evidence boundary

The re-audit report records real Docker acceptance evidence for backup and restore behavior, but exact current-HEAD full regression remains a separate validation requirement. Documentation must not claim a fresh current-HEAD regression unless that run exists and is identified by SHA and run ID.
